01 — Executive Summary

One ERP replacing six spreadsheet systems

Ormoc Printshoppe runs today on six separate Google Apps Script + Sheets tools — job orders, quotations, sales audit, large-format printing (PRISM), materials/purchasing, and task assignment — all reading and writing the same fragile shared spreadsheets. OPS Fusion replaces them with one web application, one login, one PostgreSQL database, built around the record everything already orbits: the Job Order.

Main business flow: Customer → Estimate → Job Order → Production → Costing → Delivery → Billing → Collection.

The tracking core is in progress now. The first slice — Foundation, Job Order Tracking, and Delivery Receipts — is being built through Jul–Aug, each with an automated end-to-end verify script as its definition of done, plus a legacy importer so the live sheet's data comes with it.

02 — Project Scope

What the system does

OPS Fusion tracks every customer job at per-item granularity: each line of a Job Order moves through production independently — status, department, deadline, completion — exactly matching the legacy JOWebApp behavior ("nothing less, nothing more"). Done items auto-archive and become deliverable; Delivery Receipts issue per item with partial quantities, printing on the shop's BIR-registered DR form.

Money is typed correctly everywhere (integer quantities, 2-decimal Decimal money), descriptions are never truncated, every mutation is audit-logged, and deletes are soft. The Sales side (quotations, invoices, collections, audit) is being built in parallel on a collaborator branch against the same schema — the integration points are already modeled.

03 — Development Approach

Principles that shape the build

  • Legacy rules 1:1 — "nothing less, nothing more." Every behavior is reverse-engineered from the running legacy system before it is rebuilt; no invented rules.
  • Document → then build. Each module gets a Blueprint-format design (schema, ERD, workflows, roles) approved before code — the Master System Blueprint stays the single source of truth.
  • Vertical slices. Every phase ships a complete, demoable capability — usable the day it merges, no half-built layers.
  • Verified, not just typed. Every module ships with an end-to-end verify script driving the real services against a real database — the definition of done is a green run.
  • Zero-conflict collaboration. JO-side work and Sales-side work live on separate branches over one prepared schema; integration points (advance payments, booklet numbering) are modeled and documented, not improvised.
  • Data discipline. Integer quantities, Decimal(12,2) money, no truncated text, soft deletes, one audit-log row per mutation.
  • Daily traceability. ClickUp tasks with daily comments and Drive proof links, per the Week-1 governance rules.

1

Foundation & Architecture In progress

Weeks 1–2 · Jul 7–18
Goal
Stand up the skeleton and the whole-ERP data model so every module — including the collaborator branch — builds on prepared ground.
  • Complete schema up front: 25 models covering JO, DR, and the Sales entities (Quotation, Booklet, Sale, CR, AdvancePayment, Audit) — so collaborators start on a ready schema.
  • Auth.js v5 credentials + Google login, bcrypt, role seeding.
  • CASL permission layer: 5 roles, organized Pundit-style (one policy file per resource).
  • App shell: sidebar with ability-gated navigation, theming, error envelope (ActionResult), activity log, soft deletes, counters.

Exit: login → role-appropriate shell; the whole-ERP schema migrated and ready for every later phase.

2

Job Order Tracking In progress

Weeks 2–5 · Jul 14 – Aug 8 · legacy JOWebApp 1:1
Goal
Rebuild the shop's operational heart with every legacy rule intact.
  • Per-item production tracking — each line moves independently through "Status - Department" values; timestamped status history in the legacy format.
  • Legacy rules verbatim: done-keyword auto-archive · overdue that excludes "Waiting - For Pick up / Delivery" · manual JO-number uniqueness · deadline moves with the legacy "already" guard.
  • Customer-approval gate: requires attachment proof; printable shows "THIS IS FOR APPROVAL" until approved.
  • Board metrics, calendar, archive, reports, JO PDF — and a legacy importer for the live "Line-up JOs" sheet.
  • Definition of done: a 100+ check end-to-end verify script green, typecheck + lint clean, live smoke on every page.
3

Delivery Receipts In progress

Weeks 5–7 · Aug 10–21 · replaces the paper booklet
Goal
Issue, print, edit, and track DRs against completed JO items — buildable without waiting for Sales.
  • Deliverable picker: searches done JO items with undelivered quantity; partial quantities per line, capped at remaining.
  • Full vs Partial derived from item coverage — all of a JO's items on the DR = Full; a subset = Partial (client-confirmed rule).
  • BIR-form PDF matching the physical booklet: letterhead, ATP details, Full/Partial checkboxes, cancelled watermark.
  • Metrics cards (pending deliveries, issued today/this month, partials) · edit (re-pick items, notes) · cancel returns quantities.
  • Definition of done: its own end-to-end verify suite green. Advance-payment application is modeled and deferred to the Sales merge — documented, not improvised.
4·5

Quotations · Sales & Audit Collab branch ∥

Jul–Aug · parallel track, does not consume core-dev capacity
Goal
Rebuild SignQuote and Sales-Audit on the already-prepared schema, then merge into the JO spine.
  • Quotations: inquiry log → quote builder → approval → one-click JO conversion (the Quotation ↔ JobOrder link already exists in schema).
  • Sales & Audit: booklet-numbered invoices (VAT/Non-VAT), collection receipts with full allocation ("no floating CRs"), advance payments, day-level reconciliation with locking.
  • Merge plan: schema was split per-domain with Sales models moved verbatim — porting their branch is file-scoped, and the integration TODOs (DR advance payments, booklet numbering) are marked in code.
  • Their maintenance too: Quotation Maintenance and Sales Audit Maintenance ship with these modules — the sidebar sections are already routed.
6

PRISM — Large-Format Production

Weeks 8–10 · Aug 24 – Sep 11 · after Sales-Audit · from PRISM 2.0
Goal
Fold PRISM 2.0 in: the large-format printing workflow — from JO inbox to printed roll — with substrate inventory and its own audit trail.
  • LFP JO inbox: large-format items flow straight from the JO module (replacing the legacy JO→PRISM sheet sync); auditor accepts/rejects with quantity correction and a reference-pictures gate → PJO IDs.
  • Roll & substrate inventory: LFP materials, rolls, stock-in, roll map/dashboard, roll reservations, operator open-roll requests with approval.
  • Plotting & print queue: plotting grid and layouts, per-PJO plot status, an immutable plotting log, and test prints.
  • Usage ledger + damage claims: consumption per print/test/damage; the two-step damage workflow.
  • LFP audit: pre-audit verify/flag gate before plotting; post-print audit review (audited / on-hold / discrepancy / reprint / replot); EOD report and analytics.
  • PRISM Maintenance: its own reference lists — printer registry, LFP materials, roll settings.
7

Inventory & Materials

Weeks 11–13 · Sep 14 – Oct 2 · from MACWebApp (part 1)
Goal
The materials side of the shop: what exists, where it is, and what each JO consumes.
  • Item master (codes, descriptions, UOM, locations, reorder levels, unit costs) with legacy import.
  • Stock ledger — every movement recorded; cycle counts and material adjustments with reports.
  • Material Requests per JO — request → approve → release, with stock-on-hand checks and JO history.
  • Item demand aggregation — what production needs next, before it runs out.
  • Inventory Maintenance: its own reference lists — units of measure, locations, item categories.
8

JO Task Assignment Design done

Weeks 14–15 · Oct 5–16 · from AssignedTask
Goal
Fold the AssignedTask system in: assign employees to JO lines per day and let them report progress themselves.
  • Blueprint design doc already approved-ready (schema, ERD, 6 workflows, roles matrix) — reverse-engineered from the live system.
  • Task assignment: employee × JO line × day, specific task + time segment (AM/PM/Full/OT), batch multi-day submit, deadline snapshots.
  • Employee portal: Code + PIN login (separate from staff), own-tasks only, one status update per day with an activity trail; self-reported notes.
  • Monitoring: per-employee load (loaded/light/unloaded), overdue, dept-scoped Team-Lead access.
  • Task Maintenance: its own reference lists — specific-task types, time segments, teams.
9

Purchasing

Weeks 16–17 · Oct 19–30 · from MACWebApp (part 2)
Goal
Close the supply loop: request → canvass → order → receive → stock.
  • Purchase Requests with approval flow (Pending → Approved), triggered manually or from item demand.
  • Supplier quotations per PR — canvassing captured, compared, and converted.
  • Purchase Orders → Receiving — receipts post straight into the stock ledger.
  • Supplier master and BIR-2307 groundwork.
  • Accounts Payable — the bridge to Sales & Audit: supplier invoices referencing POs and receiving, JO-linked payables, payment status, due dates and outstanding balances — with payment posting and reconciliation flowing into the Sales & Audit module.
10

JO Costing + Digital Machines

Weeks 18–19 · Nov 2–13 · the payoff phase
Goal
Answer the owner's question per job: what did it actually cost, and what did we make?
  • Machine monitoring: masterlist, usage logs, counter readings, clicks/impressions, spoilage, downtime, consumables, maintenance.
  • JO Costing — pulls from what earlier phases already record: material actuals (Phase 7), labor days (Phase 8), LFP roll usage (PRISM, Phase 6), machine usage (this phase), plus outsourcing and freight.
  • Outputs per JO: estimated vs actual cost, gross profit, margin, variance — and shop-level cost reports.

07 — Risks & Assumptions

What we're watching

  • Half-day cadence. The 4.5-month window assumes ~24 hrs/week on Fusion, alternating with A-LAB. If A-LAB demands spike, phases slip in whole-week units — flagged early via ClickUp.
  • Collaborator merge. The schema was refactored into per-domain files with Sales models moved verbatim; collaborators must rebase once, early. Integration points are already modeled to keep the merge file-scoped.
  • Sales dependency on DR. Advance-payment application and booklet numbering are deliberately deferred TODOs — they activate at the Sales merge without rework.
  • Legacy parity. Mitigated structurally: document-then-build from the running legacy systems + a per-module end-to-end verify script as the definition of done.
  • Pending client decisions. Task-module choices (PIN portal vs full users, TL department scoping) are documented as D1–D5 awaiting sign-off — building starts after approval.
  • Data migration. Live Sheets keep running until cutover; importers bring history over (the JO importer is built in Phase 2, then extended per module).
